# 性能优化阶段总结

更新日期：2026-07-17

## 1. 阶段结论

本轮性能优化已经形成一个可以独立验收的阶段成果。系统完成了工作台数据加载
拆分、历史报告索引缓存、两组安全并行、运行耗时诊断、模型调用诊断和单阶段
精确回放。多次完整分析证明并行路径可以稳定运行，同时原始串行路径仍可通过
设置或环境变量恢复。

本阶段遵守以下边界：

- 不删除分析角色，不减少辩论轮次。
- 不改变正式 Bull/Bear 交叉质询和三位风险分析师的先后依赖。
- 不以缓存替代仍应获取的最新行情。
- 性能实验默认保留回退开关，无法证明稳定收益的实验不进入正式流程。
- 模型输出长度、分析信息量和投资策略没有为了追求耗时而主动缩减。

在这些约束下，当前完整分析通常仍需二十多分钟。继续将完整分析压缩到十五分钟
以内，需要进入下一阶段，对中间输出规模、模型分工或辩论方式做受控实验。

## 2. 已完成的改动

### 2.1 工作台数据加载

- 将报告概览、最新行情和走势图从原有组合加载路径中拆分，前端并行请求相互
  独立的数据，避免慢行情阻塞历史报告展示。
- 为历史报告建立本地索引缓存。报告集合未变化时直接读取索引，不再重复扫描并
  解析所有 Markdown 报告；创建或删除报告时使缓存失效。
- 对行情和图表结果使用短期缓存，减少工作台切换标的或重复刷新时的相同请求。

这部分主要改善工作台响应速度和重复浏览体验。早期性能验证已确认缓存命中时
不会重新读取报告正文；由于当时未保留统一的浏览器基准数据，本总结不为其编造
固定毫秒级提升。

### 2.2 四个基础分析师并行

Market、Sentiment、News 和 Fundamentals 在流程上只依赖同一份初始状态，彼此
不读取对方结论，因此改为隔离分支并发执行。并行节点在合并时恢复原有报告字段
和消息顺序，并传播 LangGraph 调用配置，修复了早期工具节点缺少运行配置的问题。

默认配置：

```text
FXXKSTOCK_PARALLEL_INITIAL_ANALYSTS=true
FXXKSTOCK_PARALLEL_INITIAL_ANALYST_WORKERS=4
```

设置页可以关闭并行，关闭后恢复原始串行图。

### 2.3 Blind Bull/Bear 并行

Blind Bull 和 Blind Bear 的职责是在互不知晓对方结论的前提下独立形成首评，
天然可以并发。并行节点为两条分支复制相同输入，完成后仍按 Bull、Bear 的既定
顺序重建辩论状态，不改变后续正式交叉质询。

经过多次完整运行后，该功能改为默认开启：

```text
FXXKSTOCK_PARALLEL_BLIND_RESEARCHERS=true
```

正式 Bull/Bear 会读取上一位的回答，Aggressive、Conservative 和 Neutral 也会
读取风险辩论历史，因此这些阶段继续串行。直接并行它们会改变分析方法，不属于
本阶段的安全优化。

### 2.4 精确耗时与模型诊断

运行日志新增：

- 准备、各分析阶段、报告写入、核心要点和收尾耗时。
- 总耗时与最慢阶段汇总。
- 基础分析师和 Blind Bull/Bear 的分支耗时、并行墙钟耗时及工具轮次。
- 主要 Agent 的输入字符数、模型调用次数、结构化调用耗时、自由文本回退耗时和
  脱敏错误摘要。
- 本次实际启用的并行开关、worker 数量和证伪结构化传输方式。

这些数据同时写入报告目录的 `6_audit/audit.json`，使性能判断不再依赖终端时间戳
的人工估算。

### 2.5 单阶段精确回放

新报告在 `6_audit/replay_context.json` 保存目标 Agent 调用前的精确状态切片。
`scripts/benchmark_stage.py` 可以只回放一个昂贵阶段，不启动 Chrome、不抓取行情、
不执行整张分析图。

当前支持：

- Evidence Ledger
- 正式 Bull/Bear
- Falsification Auditor
- Trader
- Aggressive、Conservative、Neutral
- Portfolio Manager

该能力用于比较模型、结构化传输和单阶段耗时，避免每次性能实验都等待完整分析。
旧报告仍可重建近似上下文，脚本会明确标记为 `reconstructed`。

### 2.6 结构化输出与重复调用

- Falsification Auditor 在结构化结果未解析、但首次响应已有正文或 tool arguments
  时复用该响应，不再重复请求模型；由于结果未通过 schema 校验，它只作为咨询性
  审计，不会自动触发 Research Manager Revision。
- Research Manager 在首次结构化响应已有可读正文时复用正文，避免为了获得相同
  自由文本再调用一次模型。
- 证伪阶段支持 provider-aware 的 `auto` 选择。当前实测的 OpenCode DeepSeek
  路由使用 JSON mode；未知 OpenAI-compatible 网关继续使用 provider 默认值。
- Portfolio Manager 的 JSON mode 仅完成单阶段 A/B 能力，没有接入正式流程。
  两次回放均一次解析成功，但耗时分别约 55 秒和 128 秒，尚不能证明稳定更快，
  因此正式流程继续使用 `provider_default`。

### 2.7 数据与浏览器稳定性

- 东方财富 OHLCV 当天刷新失败时，可以回退到仍处于既有新鲜度阈值内的最近
  缓存，并通过同键锁避免并行分支重复刷新同一数据。
- 自动化 Chrome 默认使用后台最小化模式，也可选择 headless 或 visible。该改动
  主要消除前台抢焦点，不改变网页数据源和抓取内容。
- 本地 Web 服务默认监听 `127.0.0.1`，启动日志显示的地址可以直接访问。

## 3. 量化结果

### 3.1 同一次运行内的可比结果

`001309.SZ` 完整运行总耗时为 21 分 47 秒。由于每个并行分支都记录了自身耗时，
可以在同一股票、同一模型和同一网络环境下计算串行估算值。

| 并行组 | 分支串行合计 | 并行墙钟 | 节省时间 | 组内降幅 |
|---|---:|---:|---:|---:|
| 四个基础分析师 | 5:22 | 2:08 | 3:14 | 60.2% |
| Blind Bull/Bear | 5:10 | 2:52 | 2:18 | 44.5% |
| 合计 | 10:32 | 5:00 | 5:32 | 52.5% |

若仅把这两组恢复为串行，同一次运行的估算总耗时约为 27 分 19 秒。实际并行总耗时
为 21 分 47 秒，等价于在不改变其他阶段的情况下缩短约 5 分 32 秒，整体降幅约
20.3%。

另一份 `600839.SS` 完整运行中，两组并行分别节省约 4 分 02 秒和 1 分 38 秒，
合计约 5 分 40 秒。两份完整运行给出的直接并行收益接近，说明收益不是单次偶然值。

### 3.2 端到端观察值

加入精确计时后的早期串行样本总耗时为 26 分 48 秒；最新完整样本为 21 分 47 秒，
观察值缩短 5 分 01 秒，约 18.7%。不同样本使用了不同股票，数据量、网络响应、
模型输出长度以及是否触发 Revision 都不同，因此该数字只能说明阶段性趋势，不能
当作严格的同输入基准。

实际运行仍出现过约 25 至 29 分钟的结果。主要波动来自：

- 模型提供商响应速度变化。
- Agent 输出长度变化。
- 结构化输出失败后的第二次调用。
- Falsification 要求修订时增加 Research Manager Revision。
- 新闻、社区和公告数据源的超时与降级。

## 4. 对分析逻辑的影响

本阶段没有主动减少分析角色、辩论轮数或证据输入。两组并行只覆盖原本相互独立
的分支；正式多空质询、证伪、条件修订、交易建议、三方风险辩论和 Portfolio
Manager 仍按原顺序执行。

需要注意两项降级行为：

- 结构化结果未解析但有正文时，系统复用首次正文以避免重复调用。正文仍进入后续
  流程，但未通过 schema 的证伪结果不会自动触发修订。
- 东方财富刷新失败时只允许使用仍在既有新鲜度阈值内的缓存，并在日志中保留数据
  来源信息；不会把无限期旧数据伪装成最新行情。

## 5. 当前限制与暂缓事项

最新 `001309.SZ` 运行暴露了一个尚未处理的问题：Falsification 要求修订时，
Research Manager Revision 运行约 1 分 40 秒后结构化解析失败，最终保留初始计划。
由于修订文本未变化，旧计时逻辑又把这段时间归入 Trader。该问题同时涉及可观测性
和“证伪意见是否真正应用”的功能语义，已决定暂缓，未混入本阶段提交。

当前完整分析的大部分时间消耗在模型生成，不在 Python 计算、文件读写或 Chrome
启动。保持现有角色、信息量和串行依赖时，进一步的低风险优化预计只能将典型耗时
压到约 18 至 21 分钟。

更现实的下一阶段边界是：

- 若目标为 15 至 18 分钟，需要实验中间 Agent 的输出预算或更激进的模型分工。
- 若目标低于 15 分钟，需要把部分连续辩论改为独立并行评审，分析方式会发生变化。
- 对每天重复研究的股票，可设计真正的增量模式，目标约 7 至 12 分钟，同时保留
  当前完整分析作为定期深度研究。

上述方向均未在本阶段实施，后续需要单独评审分析质量与速度收益。
